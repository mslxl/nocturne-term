# Nocturne Agent Guide

This repository is a cross-platform terminal product targeting macOS, Linux, Windows, Android, and iOS.
The default goal is a native-feeling experience on each platform, with a polished UI, smooth motion, and a clean architecture.

## Product Intent

- Build the actual product, not a demo shell.
- Keep the first screen useful and operational.
- Favor clarity, speed, and restraint over decorative UI.

## UX Rules

- Always optimize for native feel on the current platform.
- For any desktop UI, windowing, navigation, input, or platform-integration work, proactively use the `native-feel-cross-platform-desktop` skill before making recommendations or changes.
- Prefer OS-appropriate materials, spacing, typography, focus behavior, menus, shortcuts, and window chrome.
- Motion should feel fluid, short, and responsive. Avoid animation that blocks interaction or looks ornamental.
- If a design choice conflicts with native conventions, native conventions win unless the product requirement explicitly says otherwise.
- Do not make the app feel web-first. Avoid generic browser UI patterns when a native pattern exists.

## Architecture Rules

- Keep the architecture layered: platform shell, UI layer, domain logic, and platform services should remain separable.
- Prefer thin adapters around OS-specific behavior instead of mixing platform code into core logic.
- Keep state explicit and data flow predictable.
- When doing state management, keep the state that naturally belongs in Rust in Rust, expose it through thin commands or services, and use `@tanstack/svelte-query` for async server state and cache management in the Svelte layer.
- When adding or changing Tauri command calls, derive/export the corresponding Rust and TypeScript types with `tauri-specta` so the Svelte caller stays type-safe.
- `src/lib/bindings.ts`, including the generated `commands` helpers and exported TypeScript types, is generated by `tauri-specta` during a Tauri debug-mode dev run. Do not hand-write or hand-edit this file; update the Rust command/type definitions, run the debug `dev` flow, and adapt Svelte call sites to the generated API.
- Specta forbids exporting BigInt-style Rust integer types such as `i64`, `u64`, and `usize` to TypeScript because JavaScript numbers can lose precision. For TOML/config values that need full signed 64-bit integer range, expose the value over the Tauri/Specta boundary as a decimal string and parse/validate it back to `i64` in Rust.
- Choose small, composable modules over clever abstractions.
- Reuse controls deliberately. When a UI control is used often, or is likely to be used often, proactively extract it into a reusable component with a clear, stable API.
- Preserve the existing stack unless there is a clear, documented reason to change it.

## Module Boundaries

- Avoid mixing generated bindings, page UI code, and domain logic. `src/lib/bindings.ts` remains generated; Svelte code should consume it through small feature helpers when command orchestration becomes non-trivial.

## Change Rules

- Before writing code, read the relevant material in `docs/` first to understand possible interference, impact, reusable parts, and caveats in the surrounding code.
- Read the relevant files before editing.
- Make the smallest change that solves the task.
- When extracting or introducing a reusable component, add or update a corresponding document in `docs/` that explains its purpose, API, expected usage, and native-feel caveats for future work.
- Do not refactor unrelated code while working on a request.
- Preserve user changes and unrelated worktree changes.
- If a change would affect multiple subsystems, explain the trade-off before editing.

## Quality Bar

- Verify layout, interaction, and motion on the intended platforms and sizes.
- Check dark mode, light mode, keyboard input, touch input, and narrow layouts where relevant.
- Make sure text fits its container and does not overlap neighboring content.
- Prefer failing fast over hiding contract problems with silent fallbacks.

## Implementation Discipline

- Follow the repository's existing patterns unless they are clearly wrong for the task.
- Run development, check, test, and build commands inside the Nix devShell. Prefer `nix develop -c sh -lc '<command>'` unless already inside the devShell.
- Keep async boundaries clear and errors visible.
- Add logs where they help diagnose lifecycle, platform, IPC, storage, or external-service behavior, and choose an appropriate level: `trace`/`debug` for routine diagnostics, `info` for meaningful state changes, `warn` for recoverable anomalies, and `error` for failures that require attention.
- Add tests or checks when behavior changes in a way that could regress.
- After finishing code changes, first run `pnpm check` and `cargo check` to validate frontend and backend code. Only after both pass should you attempt build checks, and all checks must pass before considering the work complete.
- If Tauri command-related types change, regenerate Specta bindings only by running the Tauri `dev` flow in debug mode, for example `pnpm tauri dev` or the repo's equivalent debug dev command. `dev` does not stop automatically, so end it manually after bindings are generated. Never manually edit generated bindings, generated command helper names, or generated TypeScript type shapes.
- Do not ship broad architectural changes without strong justification.
